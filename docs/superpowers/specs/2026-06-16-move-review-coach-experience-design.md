# MoveReview Coach Experience Design

Date: 2026-06-16

## Goal

Build a competitive MoveReview experience that feels like a coach or annotated chess book reviewing one game with the player.

The player-facing goal is not "show diagnostics." It is:

> I understand what happened in this game, why this move mattered, what I should have considered, and what I should remember next time.

Internally, the UI must remain evidence-backed. Externally, it must speak chess-player language.

## Market Research Summary

Observation:
- Chess.com Game Review wins with a clear verdict, coach guidance, show-line actions, and retry moments.
- Lichess wins with transparency and active recall, especially "learn from your mistakes."
- DecodeChess wins by structuring explanation into why/plans/threats instead of one raw engine line.
- Aimchess and Noctie win by converting review moments into future training.
- New AI-first products such as Take Take Take and Chessvia show demand for natural-language chess coaching, but also expose the risk of shallow or ungrounded AI explanation.

Inference:
- Online play, generic puzzles, opening DBs, and standard engine analysis are too crowded.
- The open space is post-game understanding: turning engine and board evidence into a coherent player lesson.
- The most defensible position is not "AI commentary"; it is "plain-language review grounded in checked chess evidence."

Hypothesis:
- Chesstory can compete by making MoveReview a guided review session: story first, move meaning second, line interaction third, evidence hidden but available.

Primary sources used during research:
- Chess.com Game Review and move classification support docs.
- Lichess analysis and learn-from-your-mistakes material.
- DecodeChess natural-language analysis and plans/threats positioning.
- Aimchess, Noctie, Maia, Chessify, Take Take Take, Chessvia, and CircleChess Caissa public product pages.

## Product Principles

1. The viewer is a chess player, not a developer.
2. The review should feel complete for one game, like a coach walkthrough or annotated game note.
3. The first screen should answer "what happened here?" before showing proof details.
4. Interactive lines should teach the board, not decorate text.
5. Internal evidence contracts stay internal. Public UI uses player language.
6. If the system cannot safely explain a cause, it should say less and show the checked line or exact fact.

## Player-Facing Information Architecture

### 1. Game Story Header

Purpose:
- Give the review a book-like opening, not a loose panel.

Content:
- Move review title from `moveReviewPlayerSurface.title`, falling back to the played SAN.
- One short story line derived from existing rendered commentary when present, without parsing it for new chess truth.
- Compact decision summary if `decisionComparison` exists.

Player language:
- `Key moment`
- `Chosen move`
- `Suggested move`
- `Difference`

Avoid:
- `diagnostics`
- `authority`
- `sourceMode`
- `support surface`

### 2. Coach Verdict

Purpose:
- Show the core decision in a quick, readable unit.

Data:
- `decisionComparison.kicker`
- `chosenSan`
- `engineSan`
- `comparedSan`
- `gapLabel`
- `secondaryText`
- `targetComparison`
- `refSans`

Behavior:
- SAN chips remain interactive only when resolved through trusted refs.
- If the chosen move matches the engine move, style the card as confirmation rather than a mistake.
- If no decision comparison exists, do not synthesize one.

### 3. Why It Mattered

Purpose:
- Elevate existing `summaryRows` from a support dump into player lessons.

Data:
- `summaryRows[].label`
- `summaryRows[].text`
- `summaryRows[].refSans`
- `summaryRows[].authority.target`
- `summaryRows[].authority.openingBook`

Presentation:
- Render 1-3 prominent coach notes first.
- Keep row text as the backend-authored claim.
- Surface target squares as board-hover chips.
- Render opening-book metadata as contextual chips.

Player language:
- `This move changed`
- `Watch this square`
- `Plan clue`
- `Opening context`

Avoid:
- Displaying authority kind names as primary text.
- Inferring a category from row text.

### 4. Try The Line

Purpose:
- Make the review interactive like an annotated game, not a static paragraph.

Data:
- Existing move chips in server-rendered commentary HTML.
- `refSans` from decision and rows.
- `refs.variations[].moves`.

Behavior:
- Reuse `renderInteractiveSanChip`, `resolveSanSequenceRefs`, and existing hover/click handlers.
- Move chips should remain compact and scan-friendly.
- If a SAN cannot be trusted to a ref, show it as display-only.

### 5. Coach Notes Body

Purpose:
- Preserve the natural-language prose and variation list generated by `MoveReviewRenderer`, but make it feel like the body of a review.

Data:
- Existing `decoded.html`.

Behavior:
- Do not parse prose to create new UI truth.
- Reposition visual hierarchy so prose supports the coach cards rather than burying the decision summary.

### 6. Deeper Review Drawer

Purpose:
- Keep transparency without making the default screen developer-facing.

Data:
- `advancedRows`
- `probeRows`
- `authorRows`

Player labels:
- `More to check`
- `Extra lines`
- `Review questions`

Behavior:
- Keep details collapsed by default.
- Use the same row renderer and interactive refs.
- Do not expose raw carrier names or diagnostics.

### 7. Lesson Ending Hook

Purpose:
- Point toward future training without overbuilding a full spaced-repetition system in the first pass.

First pass:
- Add a lightweight visible section when there are enough rows:
  - `Remember this`
  - `Replay the line`
  - `Try to find the better move next time`

Future pass:
- Save as mistake card.
- Create retry position.
- Aggregate repeated mistake themes across games.

## Implementation Boundary

First implementation pass should be frontend-first:
- Restructure `decorateMoveReviewHtml`.
- Add small presentational helpers only inside `moveReview.ts` if repeated markup demands it.
- Restyle `_side.scss`.
- Reuse all existing payload decoding and interaction code.

Do not:
- Add a parallel MoveReview runtime path.
- Reinterpret raw strategic carriers.
- Use row labels or prose to infer chess truth.
- Expose internal evidence terms as the main player UI.
- Change backend truth/trust behavior in the first pass.

## Error And Fallback Behavior

If `moveReviewPlayerSurface` is absent:
- Keep the existing commentary HTML.
- Do not render empty coach cards.

If `decisionComparison` is absent:
- Show the review title and prose first.
- Use summary rows if available.

If refs cannot resolve:
- Keep SAN display-only.
- Do not make ambiguous chips interactive.

If backend diagnostics request retry:
- Keep current retry behavior.

If a row has malformed authority:
- The decoder already downgrades authority while preserving row text.
- The redesigned UI must continue to render the text safely without target chips.

## Success Criteria

Player-facing:
- The first screen makes sense without knowing engine jargon.
- The player can immediately answer: chosen move, better move if any, why it mattered, and what line to inspect.
- The review feels like a complete guided explanation rather than a support dump.

Technical:
- Existing trusted ref behavior is preserved.
- No new backend truth path is introduced.
- Malformed or absent authority degrades safely.
- Existing MoveReview persistence still restores the rendered experience.

## Open Risks

- Existing payload may not contain enough whole-game context for a full Game Story. First pass should avoid overclaiming.
- A complete Socratic retry mode needs state and training UX beyond the current panel.
- Aggressive visual changes may expose layout issues in the constrained analysis sidebar; responsive checks are required.
- Docker Desktop may be needed for full app visual verification.
